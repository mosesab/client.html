2. What is Fortal Browser? (Project Overview)

Imagine needing a super secure way for your team to browse the internet for sensitive work. That's what Fortal Browser is for.

In simple terms, Fortal Browser provides a safe and controlled web browsing environment. It's like giving each user a private, locked-down web browser that is separate from their regular computer. This is especially useful for:

  • Sensitive Tasks: When users are working with confidential information online.
  • Collaborative Projects: Ensuring everyone in a project team is browsing in a secure and consistent way.

How does it work? Fortal Browser uses special technology called "containerization" to create isolated workspaces. Think of containers like individual, secure boxes. Each user gets their own box with a web browser inside. This box:

  • Keeps things separate: What happens in one user's browser box doesn't affect others or the main system.
  • Controls website access: We can set rules to limit which websites users can visit, ensuring they only access approved sites.
  • Adds security features: It includes built-in security measures like blocking pop-up ads and limiting keyboard shortcuts to prevent accidental actions.

The main goal of Fortal Browser is to offer a secure, focused, and easy-to-manage browsing experience for organizations that need to protect their data.

2.1. How Fortal Browser Works: The Big Picture

Let's break down how Fortal Browser works behind the scenes, step-by-step:

  1. Setting up the Brains (Server Deployment): The main part of Fortal Browser, the "brain" that manages everything, is set up on a powerful computer in the cloud. Think of this cloud computer as a central hub. We use tools to automatically install and configure Fortal Browser and its necessary database (a place to store information) on this cloud computer. We use cloud providers like Hertzner or Contabo – these are companies that rent out computer power in the cloud.

    Technical Terms Explained:

    • Cloud Server: A computer that is located in a data center (a building full of computers) and accessible over the internet. It's like renting computer space instead of owning your own server.
    • Database: An organized system for storing and managing information. In Fortal Browser, it stores user details, workspace settings, and other important data.
    • Command-line tools: Programs that you use by typing commands instead of using a graphical interface (like clicking buttons). They are often used for automating tasks on servers.

  2. Directing Traffic (NGINX Configuration): We use a tool called NGINX to act like a traffic controller for Fortal Browser. Imagine NGINX as a smart receptionist. It stands in front of the Fortal Browser system and does a few important things:

    • Secure Entry (HTTPS): It makes sure that when users connect to Fortal Browser, the connection is secure and private (using HTTPS – like a security guard for internet traffic).
    • Traffic Management (Reverse Proxy): It efficiently directs incoming requests from users to the right parts of the Fortal Browser system.

    Technical Terms Explained:

    • NGINX (Reverse Proxy): A software that sits in front of web applications and manages incoming traffic. It improves security, performance, and reliability. Think of it as a gatekeeper for the system.
    • HTTPS: "Hypertext Transfer Protocol Secure". It's the secure version of HTTP, the protocol used to transfer data over the web. HTTPS encrypts communication, making it secure from eavesdropping. You can recognize it by the padlock icon in your browser's address bar.

  3. Security Certificates (SSL Implementation with Certbot): To make sure the connections are secure (HTTPS), we need special digital certificates called SSL certificates. We use a tool called Certbot to automatically get and set up these certificates for free. This is like getting an official "security badge" for our website, proving it's trustworthy and secure.

    Technical Terms Explained:

    • SSL Certificate: "Secure Sockets Layer" certificate. A digital certificate that verifies a website's identity and enables encrypted connections (HTTPS). It's essential for secure communication over the internet.
    • Certbot: A free, automated tool that makes it easy to get and renew SSL certificates. It simplifies the process of enabling HTTPS on websites.

  4. Isolated Browsers (Docker and Firefox Profiles): We use a technology called Docker to create those isolated "browser boxes" we talked about. Docker lets us package up the Firefox browser and everything it needs to run in a self-contained unit (a "container"). We also customize Firefox profiles – these are like settings files for Firefox – to add extra security and features to these browser boxes.

    Technical Terms Explained:

    • Docker: A platform that allows you to create, run, and manage applications in "containers." Containers are like lightweight virtual machines that isolate applications from each other and the host system, ensuring consistency and portability. Think of it as packaging an application and its dependencies into a box that can run anywhere.
    • Firefox Profiles: Sets of configurations and customizations for the Firefox web browser. Profiles allow you to have different settings, extensions, and bookmarks for different browsing environments. In Fortal Browser, custom profiles enhance security and functionality.

3.2. Key Components: What Each Part Does

  • User Interface (UI): What you see and use
    • What it is: The screens and buttons you interact with to use Fortal Browser. It's built using web technologies like HTML, CSS, and JavaScript.
    • Why it's important: This is how users access and control workspaces. It needs to be user-friendly and secure.
    • How it works:
      • client.html: This is the main user portal where users log in and access their assigned workspaces. It's hosted on vercel.app (a platform for hosting websites).
      • admin-client.html: This is the admin area. Only administrators can access this to manage users and workspaces. It's also hosted on vercel.app.
      • preview-client.html: A special version of the user interface used for testing new features before they are released to all users.
  • Reverse Proxy (Nginx): The Traffic Manager
    • What it is: As explained earlier, NGINX acts as the entry point for all web traffic to Fortal Browser.
    • Why it's important: It secures connections, manages traffic efficiently, and directs requests to the right place.
    • How it works:
      • SSL Security: It handles the secure HTTPS connections.
      • Traffic Routing: It directs user requests to the correct browser container.
      • Configuration: Its settings are stored in files located at /etc/nginx/sites-available/default on the server.
  • Application Server (FastAPI with uvicorn): The Brains of the Operation
    • What it is: This is the core of Fortal Browser. It's a program that handles all the important logic and tasks. It's built using Python (a programming language) and tools called FastAPI and Uvicorn.
    • Why it's important: It manages users, workspaces, file sharing, and all the main functions of Fortal Browser.
    • How it works:
      • Logic and Management: It contains the rules and processes for how Fortal Browser operates.
      • Data Storage: It communicates with the database to save and retrieve information.
      • Location: The code for this server is located in the /root/api directory on the server.
  • Containerization (Docker): The Isolation Boxes
    • What it is: Docker is used to create and manage isolated containers for each browser workspace.
    • Why it's important: It ensures each workspace is separate and consistent, improving security and stability.
    • How it works:
      • Isolation: Each browser runs in its own container, separate from other workspaces and the main server.
      • Consistency: Docker ensures every workspace starts from the same base setup.
      • Building Containers: Docker uses special files called Dockerfile (located in the main project directory) to build these container images.
      • Using noVNC: It uses a pre-built container image called "noVNC" which helps display the browser in the user's web browser.
  • Database (SQLite): The Information Storage
    • What it is: A database is used to store all the important information for Fortal Browser. In this case, it's SQLite, a simple and file-based database.
    • Why it's important: It stores user accounts, workspace settings, access rules, and other system data that needs to be remembered.
    • How it works:
      • Storing Data: It saves all the necessary information in a structured way.
      • File Location: The database file is named app.db and is created in the project directory if it doesn't already exist.
  • Firefox Browser: The User's View
    • What it is: The actual web browser that users interact with. It's Firefox, but running inside the Docker container on the server.
    • Why it's important: This is the tool users use to browse the web securely.
    • How it works:
      • Remote Browser: The browser runs on the server, not the user's computer. What users see is a visual stream of the browser window.
      • noVNC Client: The noVNC technology is used to display the remote browser in the user's web browser.
      • Customized Firefox: Firefox is modified with special settings and add-ons to enhance security and control.
  • Custom Firefox Profiles: Special Settings for Security and Control

    These are special configurations applied to Firefox to make it more secure and controlled:

    
*   **Add-ons and Extensions**:
    *   This folder contains extra features (add-ons and extensions) that are automatically installed in the Firefox browser to enhance its functionality.

*  **userChrome Customization:**
    * The  `userChrome.css` file is used to change how Firefox looks and feels.  We use it to:
    * Hide unnecessary buttons and menus (like the bookmarks bar) to make the browser simpler and less cluttered.
    * Change the browser's appearance to improve usability.

*  **Configuration Settings with prefs.js:**
    *  The  `prefs.js` file is used to set specific preferences for Firefox to improve security and control. We use it to:
    *  Disable developer tools (which are used for advanced web development) to reduce the risk of unauthorized changes or attacks.
    *  Turn off automatic updates and data sharing (telemetry) to maintain a consistent and private browsing environment.

3.3. Container Registry: Where Browser Images Come From

The "blueprint" for creating the browser containers (Docker images) is stored and managed. For Fortal Browser, these images are pulled from Docker Hub (a public repository of container images).

If you need to rebuild the browser image, you would use this command in the project directory:

docker build -t novnc .

Technical Terms Explained:

  • Docker Hub: A service for finding and sharing container images. It's like an app store for Docker containers.
  • Docker Image: A template or blueprint used to create Docker containers. It contains everything needed to run an application, including the code, libraries, and settings.
  • docker build -t novnc .: This is a command that tells Docker to build a new container image. novnc is the name we are giving to the image, and . means to use the Dockerfile in the current directory.

3.4. APIs: How the Parts Communicate

APIs (Application Programming Interfaces) are like messengers that allow different parts of Fortal Browser to talk to each other. We use RESTful APIs, which is a standard way for software systems to communicate over the internet.

Here are some important API "endpoints" (specific addresses for sending messages) that Fortal Browser uses:

  • /api/workspaces/*: These are for managing workspaces. You can use these to create new workspaces, list existing ones, and shut down workspaces.
  • /api/superadmin/*: These are special endpoints only for super administrators. They are used for managing administrators themselves and overseeing all workspaces on the server.
  • /api/chats/*: These handle the chat messaging feature within workspaces.
  • /api/files/*: These are for uploading, downloading, and managing files within workspaces.
  • /api/check/*: These provide information like the number of files and chat messages.

Technical Terms Explained:

  • API (Application Programming Interface): A set of rules and specifications that software programs can follow to communicate with each other. APIs allow different parts of a system to work together. Think of it as a menu in a restaurant – it lists the available "requests" (API calls) you can make and what "responses" (data) you will get back.
  • RESTful APIs: A popular style of API design that uses standard web protocols (like HTTP) for communication. They are widely used because they are simple, scalable, and easy to understand.
  • Endpoint: A specific URL (web address) where an API "listens" for requests. Each endpoint is designed to handle a specific type of request and perform a particular action.

3.5. Data Flow: How it Works Step-by-Step

Here's a step-by-step look at what happens when a user starts a new workspace:

  1. User Request: You (a user) use the User Interface (UI) to request a new workspace.
  2. UI to Application Server: The UI sends a message (an API request) to the Application Server asking to start a workspace.
  3. Authentication and Database Update: The Application Server checks if you are authorized to create a workspace. If you are, it records the new workspace request in the database.
  4. Starting the Container: The Application Server tells Docker to create and start a new browser container based on the pre-defined image.
  5. Nginx Configuration: Nginx is automatically updated to direct web traffic to this new container.
  6. Workspace Connection: You connect to the new workspace through the UI. The UI communicates with the noVNC software inside the container to display the browser on your screen.

4.2. Step-by-Step Setup Guide

Follow these steps to set up Fortal Browser on your server:

  1. Get the Fortal Browser Files (Clone the Repository):

    First, you need to download the Fortal Browser software files from where they are stored (a "repository"). Use these commands in the server's command line:

    git clone   # Replace  with the actual web address of the repository
cd  # Replace  with the name of the folder created after cloning

    Technical Terms Explained:

    • Git: A system for tracking changes in code and collaborating on software projects. git clone is a command to download a copy of a repository.
    • Repository: A storage location for code and other files, usually managed by Git. Think of it as a project folder with version control.
    • cd: "Change Directory". A command in the command line to navigate to a different folder.

  2. Build the Browser Container (Build the Workspaces Docker Container):

    Next, you need to build the Docker image for the browser workspaces. Navigate to the firefox directory and run the Docker build command:

    cd firefox && ls # Go into the 'firefox' folder and list its contents (ls - list)
docker build novnc . # Build the Docker image named 'novnc' using the Dockerfile in the current folder

    Technical Terms Explained:

    • &&: In command line, this means "run the next command only if the previous one was successful".
    • ls: "List". A command to list the files and folders in the current directory. It's used here to check the contents of the firefox directory.
    • docker build novnc .: As explained before, this command builds a Docker image.

  3. Create a Folder for Admin Files (Create Necessary Directories):

    Create a directory for admin files and set the correct permissions. Use these commands:

    sudo mkdir /home/ftpuser/admin_fortal_files # Create a directory named 'admin_fortal_files'
sudo chown ftpuser:ftpuser /home/ftpuser/admin_fortal_files # Change ownership of the directory to the user 'ftpuser'

    Technical Terms Explained:

    • sudo: "Super User Do". A command to run another command with administrator privileges.
    • mkdir: "Make Directory". A command to create a new folder.
    • chown: "Change Owner". A command to change the owner and group of a file or directory.
    • ftpuser:ftpuser: Specifies the new owner and group for the directory. In this case, both are set to ftpuser.

  4. Install Python Libraries (Install Python Packages):

    Install the necessary Python libraries that the Application Server needs. Run this command:

    pip3 install --break-system-packages requests aiodocker fastapi uvicorn pydantic httpx python-multipart websockets

    Technical Terms Explained:

    • pip3 install: A command to install Python packages using the pip package installer.
    • --break-system-packages: A flag sometimes needed to avoid conflicts with system-level Python packages (use with caution, and only if needed as per setup instructions).
    • requests, aiodocker, fastapi, uvicorn, etc.: These are names of specific Python libraries (collections of code) that Fortal Browser uses.

  5. Start the Admin Dashboard (Running app.py):

    Start the Application Server (Admin Dashboard) by running the app.py Python script:

    ```python
python3 api/app.py

    Technical Terms Explained:

    • python3 api/app.py: This command runs the Python script app.py, which is located in the api directory. This starts the Application Server.

  6. Configure Nginx (Configure Nginx):

    Set up Nginx to handle web traffic for Fortal Browser:

    • Remove Default Nginx Configuration:

      sudo rm /etc/nginx/sites-enabled/default # Remove the default Nginx configuration file

      Technical Terms Explained:

      • rm: "Remove". A command to delete files or directories.
      • /etc/nginx/sites-enabled/default: The location of the default Nginx configuration file that we need to remove to replace it with our own.

    • Copy the Fortal Nginx Configuration:

      sudo mv default.txt /etc/nginx/sites-available/default # Move the 'default.txt' file to the Nginx configuration directory and rename it to 'default'

      Technical Terms Explained:

      • mv: "Move". A command to move or rename files or directories.
      • default.txt: The file containing the Nginx configuration for Fortal Browser.
      • /etc/nginx/sites-available/default: The standard directory in Nginx where configuration files for websites are stored.

    • Enable the Fortal Site:

      sudo ln -s /etc/nginx/sites-available/default /etc/nginx/sites-enabled/ # Create a symbolic link to enable the Nginx configuration

      Technical Terms Explained:

      • ln -s: "Link - Symbolic". A command to create a symbolic link (a shortcut) to a file or directory.
      • /etc/nginx/sites-enabled/: The directory in Nginx where symbolic links to enabled website configurations are placed.

    • Restart Nginx to Apply Changes:

      sudo systemctl restart nginx # Restart the Nginx service to load the new configuration

      Technical Terms Explained:

      • systemctl restart nginx: A command to restart the Nginx service, which makes it reload its configuration files and apply the changes.

  7. Set Up Secure Connections (Set Up SSL):

    Enable HTTPS for secure connections using Certbot:

    sudo apt install certbot python3-certbot-nginx # Install Certbot and its Nginx plugin
sudo certbot --nginx -d workspaces.fortal.io # Run Certbot to get and configure SSL certificate for 'workspaces.fortal.io' using Nginx plugin
sudo certbot install --cert-name workspaces.fortal.io # Install the SSL certificate (this line might be redundant, Certbot usually installs automatically)

    Technical Terms Explained:

    • sudo apt install certbot python3-certbot-nginx: Installs Certbot and the Nginx plugin for Certbot using the apt package manager (common on Debian/Ubuntu Linux).
    • certbot --nginx -d workspaces.fortal.io: Runs Certbot, telling it to use the Nginx plugin and to get a certificate for the domain workspaces.fortal.io (replace with your actual domain name).
    • certbot install --cert-name workspaces.fortal.io: This command is likely redundant as Certbot usually installs the certificate automatically. If needed, it would explicitly install the certificate named workspaces.fortal.io.

  8. Schedule Automated Tasks (Set Up Automated Tasks (Cron)):

    Set up automated tasks using cron to perform regular maintenance, like cleaning up old Docker data. Run this command to edit the cron schedule:

    crontab -e # Edit the cron table (schedule of automated tasks)
# Run every Sunday at midnight
0 0 * * 0 docker system prune -af --volumes # This line, when added to crontab, will run the Docker cleanup command every Sunday at midnight

    Technical Terms Explained:

    • crontab -e: Opens the cron table file for editing. Cron is a system in Linux to schedule commands to run automatically at specific times or intervals.
    • 0 0 * * 0 docker system prune -af --volumes: This is a line added to the cron table.
      • 0 0 * * 0: This part defines the schedule: run at minute 0, hour 0 (midnight), every day of the month, every month, and on day 0 of the week (Sunday).
      • docker system prune -af --volumes: This is the command that will be run on the schedule. It's a Docker command to remove unused Docker data (like old containers, images, and volumes) to free up disk space.

  9. Run the Application Server in the Background (Running app.py in background):

    To keep the Application Server running even after you close your terminal session, use tmux (a terminal multiplexer):

    • Install tmux:

       sudo apt install tmux # Install tmux

    • Start a tmux Session and Run app.py:

       (tmux kill-session -t flask_terminal) # Optionally stop any existing tmux session named 'flask_terminal'
tmux new-session -s flask_terminal # Create a new tmux session named 'flask_terminal'
python3 app.py # Run the Application Server script
Detach terminal session and leave it running perpetually:
Ctrl + B, then D (just d). # Press Ctrl+B, then D to detach from the tmux session (it will keep running in the background)
Run tmux in copy mode (for scrolling):
Ctrl + B, then [ (just [). # Press Ctrl+B, then [ to enter copy mode in tmux, allowing you to scroll through the terminal output
tmux attach-session -t flask_terminal # Re-attach to the tmux session named 'flask_terminal' to see its output again
tmux ls # List active tmux sessions

      Technical Terms Explained:

      • tmux: "Terminal Multiplexer". A tool that allows you to run multiple terminal sessions within a single window. It also allows you to detach and re-attach to sessions, so processes can continue running even if you close your terminal window.
      • tmux kill-session -t flask_terminal: Kills (stops) a tmux session named flask_terminal if it exists. This is optional, to start fresh.
      • tmux new-session -s flask_terminal: Creates a new tmux session and names it flask_terminal.
      • Ctrl + B, then D: A shortcut in tmux to detach from the current session.
      • Ctrl + B, then [: A shortcut in tmux to enter copy mode, which allows you to scroll up and down in the terminal output.
      • tmux attach-session -t flask_terminal: Re-attaches to an existing tmux session named flask_terminal, allowing you to see its output and interact with it.
      • tmux ls: Lists all currently active tmux sessions.

5.6. Backups and Recovery: Protecting Your Data

Regular backups are crucial to protect against data loss.

  • Database Backups:

    • Why: The database (app.db or MySQL, clarify which is used) stores important data like user accounts and workspace settings. Backing it up regularly prevents losing this information if something goes wrong with the server.
    • How:
      • Schedule Regular Backups: Set up a schedule for automatic database backups (e.g., daily or weekly).
      • Backup Script: You can create a simple script that copies the database file (app.db if using SQLite) or uses MySQL backup tools (like mysqldump if using MySQL) to a safe location.   • Container Data:

    • Why: If workspaces store important data within the containers, you need to consider backing up this data as well.
    • How:
      • Persistent Volumes: Docker allows you to use "persistent volumes" to store container data outside of the container itself. If persistent volumes are used for workspaces, you should back up these volumes.
      • Backup Strategy: Develop a backup strategy for container data if it's critical. This might involve backing up Docker volumes or using container-aware backup solutions.

6. Monitoring & Maintenance: Keeping Fortal Browser Healthy

Regular monitoring and maintenance are essential to ensure Fortal Browser runs smoothly and securely.

6.1. Monitoring: Checking System Health

Keep an eye on these things to make sure Fortal Browser is working correctly:

  • Container Status:
    • What to check: Are the browser containers running?
    • How to check: Use the command docker ps in the server's command line. This will show a list of running Docker containers and their status. You should see containers related to Fortal Browser workspaces listed as "running".
    • What to look for: Ensure the containers are listed and their status is "Up" or "Running". If any containers are missing or showing an error status, it indicates a problem.
  • Resource Usage:
    • What to check: How much CPU (processing power), memory (RAM), and network bandwidth are the containers using?
    • How to check: Use the command docker stats in the server's command line. This will show real-time resource usage for each container.
    • What to look for: Check for unusually high CPU or memory usage by any container. High usage might indicate a problem or a workspace consuming excessive resources.
  • Application Logs:
    • What to check: Are there any errors or warnings in the logs of the Application Server?
    • How to check: Check the application log file, which is mentioned as /tmp/app.log. You can view this file using command-line tools on the server (e.g., cat /tmp/app.log or less /tmp/app.log).
    • What to look for: Look for lines in the log file that indicate errors ("ERROR") or warnings ("WARNING"). These messages can provide clues about problems in the Application Server.
  • Nginx Logs:
    • What to check: Are there any errors or unusual access patterns in the Nginx web server logs?
    • How to check: Check the Nginx log files, usually located in /var/log/nginx/. There are typically two main logs: access.log (records all requests) and error.log (records errors). You can view these files using command-line tools.
    • What to look for: In error.log, look for error messages. In access.log, look for unusual patterns or a large number of errors (e.g., HTTP status codes like 404 Not Found or 500 Server Error).
  • Container Logs (Detailed Logs inside Containers):
    • What to check: For more detailed logs from inside the browser containers themselves.

7. Troubleshooting & FAQs: Solving Common Problems

This section provides solutions to common issues and answers frequently asked questions.

7.1. Common Issues and Solutions

  • Cannot Access Workspace:
    • Possible Causes: You try to access a workspace, but it doesn't load or you get an error message.
    • Troubleshooting Steps:
      • Check Container Status: Use docker ps to see if the workspace container is running. If it's not running, there's likely a problem with the container.
      • Check Nginx Configuration and Logs: Review the Nginx configuration (/etc/nginx/sites-available/default) and Nginx logs (/var/log/nginx/error.log) for any errors or misconfigurations that might be preventing access.
      • Firewall Rules: Ensure that the server's firewall is configured to allow traffic to the port that the workspace container is using.
      • Docker Instance Running: Verify that the Docker service itself is running correctly on the server. You can check Docker's status using commands like sudo systemctl status docker.
  • Workspace is Unresponsive (Slow or Freezing):
    • Possible Causes: A workspace starts, but it becomes slow, unresponsive, or freezes.
    • Troubleshooting Steps:
      • Check Container Resource Usage: Use docker stats to see if the container is using too much CPU or memory. High resource usage can cause unresponsiveness.
      • Restart the Container: Try restarting the workspace container using docker restart (replace with the actual container ID from docker ps). Restarting can sometimes resolve temporary issues.
      • Investigate Application Logs: Check the Application Server logs (/tmp/app.log) for any errors or performance issues that might be affecting workspace responsiveness.
  • File Uploads Failing:
    • Possible Causes: You try to upload a file to a workspace, but the upload fails.
    • Troubleshooting Steps:
      • File Size Limits: Check if there are any file size limits configured for uploads. If the file is too large, it might be rejected.
      • Directory Permissions: Verify that the user account within the workspace has the necessary permissions to write files to the upload directory. Check file permissions on the server if needed.
      • Disk Space: Ensure there is enough free disk space on the server where files are being uploaded.
  • "New Workspace" Keeps Loading, But Never Loads:
    • Possible Causes: You click "New Workspace," but it gets stuck loading and never starts.
    • Troubleshooting Steps:
      • Check Docker Containers: Use docker ps -a (to show all containers, running and stopped). Ensure that there aren't any old, stopped workspace containers that might be interfering. Clean up stopped containers if necessary (using docker container prune or similar commands).
      • Server Resources: Check if the server is overloaded on resources (CPU, memory, disk space). If the server is struggling, it might not be able to start new containers.
      • Application Server Logs: Check the Application Server logs (/tmp/app.log) for errors that might be occurring when trying to create a new workspace.

8. Security Considerations: Keeping Fortal Browser Secure

Security is paramount for Fortal Browser. Here are important security practices to follow:

  • Password Management:
    • Strong Passwords: Use strong, unique passwords for all accounts (user accounts, admin accounts, server access).
    • Password Rotation: Implement a policy for regularly changing passwords, especially for administrator accounts.
    • Secure Storage: Store passwords securely. Avoid storing passwords in plain text.
  • Regular Updates:
    • Keep Everything Updated: Regularly update the server's operating system, Docker, Nginx, and all other software components. Updates often include critical security patches.
    • Automate Updates (Where Possible): Use automated update mechanisms where appropriate to ensure timely patching.
  • Access Control:
    • Restrict Access: Limit access to the server and the admin interface to only authorized personnel. Use strong authentication methods (like SSH keys for server access).
    • Principle of Least Privilege: Grant users and administrators only the minimum level of access they need to perform their tasks.
  • Firewall Configuration:
    • Configure Firewalls: Set up firewalls on the server to allow only necessary network traffic. Block all unnecessary ports and services.
    • Network Segmentation: If possible, segment the network to isolate the Fortal Browser server and containers from other less secure systems.
  • HTTPS Certificates:
    • Use HTTPS Everywhere: Ensure that HTTPS is enabled for all communication with Fortal Browser (both user access and API communication). This encrypts data in transit.
    • Valid Certificates: Use valid SSL/TLS certificates from a trusted Certificate Authority (like Let's Encrypt, which Certbot helps with).
  • Data Encryption:
    • Encryption in Transit (HTTPS): HTTPS, as mentioned, encrypts data while it's being transmitted over the network.
    • Encryption at Rest (Database): If using MySQL (as per prerequisites), ensure that data at rest in the database is also encrypted. MySQL supports encryption features. (If using SQLite, consider encryption options for SQLite databases as well, if sensitive data is stored).
